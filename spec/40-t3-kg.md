# T3 — knowledge graph

T3 adds a second, independent view of the brain: an entity/relation graph
alongside the doc graph. Where T1 records the pages authors wrote and linked,
T3 records the concepts *between* the pages. Two backends produce it:

- **algorithmic** (the default): derived deterministically from what the
  files already carry — ghosts (concepts referenced but not yet written) and
  tags. Needs nothing: no model, no endpoint. Its export is byte-reproducible
  and therefore conformance-golden.
- **lightrag** (opt-in, `brainpick[graph]`): an LLM extracts entities and
  relations from the prose — richer on corpora whose structure is not yet
  encoded in links, at model cost. Extraction is not deterministic, so its
  export content is advisory.

Whatever the backend, this tier is split the same way:

- The **neutral export** (`t3/entities.jsonl`, `t3/relations.jsonl`,
  `t3/kg-meta.json`) has a **normative layout**: every consumer — the Node
  engine, the UI's entity layer, `mode=graph` search, `brain_neighbors
  layer=entities` — reads only these files, never the extractor's private
  working directory. Field names, id normalization, canonical ordering, and
  the query semantics over the export are normative.
- The export **content** is advisory for LLM backends (two runs may disagree
  on which entities exist — consumers are conformance-tested against a
  hand-authored fixture) and **normative for the algorithmic backend** (the
  derivation below is exact, so both engines MUST produce byte-identical
  exports, proven by goldens).

## Backends are private; the export is the product

Every backend is a `KGBackend` behind the same neutral export; nothing
downstream knows which one ran.

- **algorithmic** derives the export from T1 data (below). It is pure
  computation, so it runs **natively in both engines** — no delegation, no
  extras, no endpoint.
- **lightrag** runs LightRAG (Python-only, `brainpick[graph]`) in
  `t3/lightrag/` — an opaque working directory outside this spec; the Python
  compiler translates its output into the neutral export. The Node engine
  does not extract: with `graph = "lightrag"` its T3 compile delegates to an
  installed Python sibling (`uvx --from 'brainpick[graph]' brainpick compile
  --only t3`) or skips with an instruction. T3 **query** over the export is
  native in both engines regardless of backend.

`[modules] graph` selects: `"algorithmic"` (default) | `"lightrag"` |
`"auto"` (lightrag when `[models.extraction]` is configured, else
algorithmic) | `"off"` (spec/80).

## The algorithmic backend (normative derivation)

The default backend encodes this spec's philosophy: authors (human or LLM)
create connections proactively as links and tags; the knowledge graph is then
*derived*, never extracted. Its entities are exactly the concepts the wiki
references but does not have pages for:

- **Ghost entities** — one per distinct link target that does not exist in
  the bundle: `id` = the target path's stem, normalized (the same id rules
  below); `name` = the link TEXT of the first reference, taking docs in
  sorted-path order and references in document order (link text is the
  concept's title by OKF convention); `type` = `"ghost"`;
  `description` = `"Referenced from N page(s) but not yet written."` (exact,
  N = source-doc count); `source_docs` = the referencing docs, sorted.
- **Tag entities** — one per distinct tag: `id` = the tag normalized;
  `name` = the tag as first written (sorted-path order); `type` = `"tag"`;
  `description` = `"Tagged on N page(s)."`; `source_docs` sorted.
- **Relations — co-occurrence**: one per unordered entity pair sharing at
  least one source doc. `src` < `dst` by id; `weight` = `1 − 2^(−shared)`
  where `shared` = the number of shared docs (exactly representable:
  0.5, 0.75, 0.875, …); `description` = `"Co-mentioned in N page(s)."`;
  `keywords` = `[]`; `source_docs` = the shared docs, sorted.
- `kg-meta.json`: `{"entities": N, "extractor": {"kind": "algorithmic"},
  "relations": M, "spec_version": "0.1"}`.

An **empty export is valid and fresh**: a fully-written, untagged wiki
genuinely has no sub-page concepts — consumers (UI entity layer, mode=graph)
MUST tolerate zero entities. Serialization is canonical (spec/10); the whole
export is byte-golden per fixture.

## t3/entities.jsonl (normative layout, advisory content)

One canonical JSONL line per entity, sorted by `id`:

```json
{"description":"The star everything orbits.","id":"aurinko","name":"Aurinko","source_docs":["aurinko.md","planeetat.md"],"type":"star"}
```

- `id` — the entity's stable key: the `name` normalized (NFC, lowercased,
  every run of non-alphanumeric characters → `-`, trimmed of `-`). Two
  extractions that name the same entity converge on the same id, which keeps
  the entity layer's [live deltas](live-deltas.md) quiet across recompiles.
  Collisions (distinct names, same slug) are disambiguated `-2`, `-3`… in
  `name` codepoint order.
- `type` — a short extractor-chosen descriptor (`star`, `person`, `concept`);
  free-form, lowercased. `description` — one sentence. `source_docs` — the
  bundle paths whose chunks mention the entity, sorted, deduplicated.

## t3/relations.jsonl (normative layout, advisory content)

One line per directed-or-undirected relation, sorted by (`src`, `dst`):

```json
{"description":"orbits","dst":"aurinko","keywords":["orbit","gravity"],"src":"maa","source_docs":["maa.md"],"weight":0.8}
```

`src`/`dst` are entity ids (must exist in `entities.jsonl`). `weight` is a
float in [0,1]. `keywords` is a sorted, deduplicated string list. Relations
are stored once per unordered pair with the extractor's chosen direction; a
consumer treats the graph as undirected for neighbor walks.

## t3/kg-meta.json (normative)

```json
{"entities": 12, "extractor": {"kind": "lightrag", "model": "qwen3.6:35b"},
 "relations": 9, "spec_version": "0.1"}
```

## Entity vectors (optional)

When T2 is on, the exporter MAY embed each entity's `name + " — " +
description` into a LanceDB `entities` table (columns `id`, `name`, `vector`),
using the same embedding record as [T2](30-t2-vectors.md). This powers
semantic entity search; absent it, entity search falls back to keyword over
`entities.jsonl`. Advisory — consumers tolerate its absence.

## Retrieval over the export (normative semantics)

- `brain_neighbors({doc, layer: "entities"})` — resolve the doc's entities
  (those whose `source_docs` include it), then walk `relations.jsonl` to
  depth N; return entities with descriptions and the `source_docs` that
  ground each. `layer: "both"` overlays the T1 link neighbors and the entity
  neighbors, tagged by layer.
- `mode=graph` search (spec/50): match the query against entity names and
  descriptions, expand one relation hop, and rank the `source_docs` of the
  matched-and-adjacent entities — answering "what connects to X" rather than
  "what says X". With T3 off, `graph` degrades to a T1 link-graph walk over
  keyword hits, `degraded_from: "graph"`.

## Tier status

`tiers.t3` in the manifest: `fresh` after a backend pass over the current
chunk set, `stale` when chunks changed since, `off` only when
`[modules] graph = "off"` (the algorithmic default needs no endpoint, so an
unconfigured brain has a fresh — possibly empty — T3). Extraction runs last and never blocks T1/T2 (spec/00). It
reuses the [compile pipeline](../docs/compile-pipeline.md) change set:
only changed docs' chunks are re-extracted (the reference extractor supports
incremental insert), and `id` normalization keeps unchanged entities stable.

The tier must reflect what is on disk: a compile MUST NOT report `fresh` when
no export (`t3/entities.jsonl`) is present. An export deleted out of band forces
a rebuild — re-extracted where the extractor is reachable, else degraded to
`stale` — and turning the tier off (or removing the export) makes a subsequent
compile record `off`, so `/api/status` and the report never overstate T3. An
engine that only *reads* the export (the Node engine, which never extracts)
reports `fresh` while a current export is staged and `off` once it is gone.

## Conformance

Class `kg-query` (both engines, zero skips): a hand-authored export fixture
under `spec/fixtures/expected/<bundle>/t3/` is the input; cases assert the
document SET returned by `brain_neighbors layer=entities` and `mode=graph`
search over it. LLM extraction is never invoked in conformance — only the
normative reader is. Id normalization has its own golden (names → ids).

Class `kg-algorithmic` (both engines, zero skips): compile the fixture bundle
with `graph = "algorithmic"` and compare the produced
`entities.jsonl`/`relations.jsonl`/`kg-meta.json` byte-for-byte against
goldens (regenerated only via `scripts/regen-golden.py`). The derivation is
deterministic, so the algorithmic T3 is held to the same byte-golden standard
as T1.
